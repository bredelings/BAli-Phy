# RE/flex Lexer Plan

This plan starts from the current lexer architecture, where Flex still
recognizes raw source tokens but Haskell layout insertion is handled by the
parser-facing driver wrapper.

The next switch should be an ASCII-equivalent raw-scanner conversion.  Unicode
support should come later, after the RE/flex scanner is structurally clean and
covered by tests.

## Current State

The Haskell lexer now has three distinct responsibilities:

```text
source text
  -> raw scanner
  -> driver wrapper that inserts virtual tokens
  -> parser
```

The raw scanner still lives in `src/computation/parser/lexer.l` and is
generated by Flex.  It recognizes real source tokens, skips whitespace and
comments, handles scanner states for nested comments and pragmas, decodes
literals, and records token metadata such as `starts_line`.

The driver wrapper lives in `driver::next_parser_token()`.  It may hold one
pending real token, return queued virtual tokens, perform beginning-of-line
layout checks, handle post-keyword layout, and commit real token effects only
when the parser receives that token.

The old layout-specific Flex empty rules and states have been removed:

- `<bol>""/.`
- `<layout>""/.`
- `<layout_do>""/.`
- `<layout_if>""/.`
- `<layout_left>""/.`
- `bol`, `layout`, `layout_do`, `layout_if`, and `layout_left` start states

This means the RE/flex conversion should not need layout probe rules,
`yyless(0)` rollback, recursive `yylex(drv)` calls for layout, or
layout-specific start states.

## Remaining Flex Dependencies

The current raw scanner still depends on Flex-specific interfaces:

- `YY_DECL`
- `YY_USER_ACTION`
- `yy_scan_string`
- `yy_delete_buffer`
- `YY_CURRENT_BUFFER`
- `yy_flex_debug`
- `%option stack`
- `%s` start states for nested comments, pragmas, and line pragmas
- `yy_push_state()` and `yy_pop_state()`
- the `yy_top_state()` compatibility reference used to silence generated-code
  warnings

These are ordinary scanner mechanics now, not Haskell layout mechanics.  They
should be replaced by RE/flex scanner-object APIs during the raw-scanner
conversion.

## Build Integration

The previous experiment used a temporary `/tmp/re-flex` installation.  A
mergeable conversion needs a real build design.

Open questions:

- Can RE/flex be configured as a Meson subproject through its CMake build?
- Should the build prefer a system RE/flex dependency, a wrap, or both?
- Does the scanner mode we want require linking the RE/flex runtime library?
- Is the RE/flex license acceptable for BAli-Phy distribution?
- Should generated scanner sources remain checked in, as they are for Flex?
- How should Linux, macOS, and native Windows CI obtain the generator and any
  runtime library?

The build design should be settled before changing the maintained scanner
source.  A temporary local RE/flex path is acceptable only on an experiment
branch.

## Raw Scanner Conversion

The first RE/flex scanner should preserve current ASCII behavior.

Keep these pieces unchanged during the first conversion:

- `driver::next_parser_token()`
- virtual token insertion
- layout context stack behavior
- `LexedToken` and `TokenEffects`
- committed-source-token adjacency logic
- parser grammar behavior

Replace only the raw scanner implementation and scanner lifetime management:

- instantiate and own an RE/flex scanner object with a clear parse lifetime;
- reset scanner state at the start of each parse;
- install the input buffer/string through RE/flex APIs;
- map scanner debug settings from `driver::trace_scanning`;
- replace Flex start-state calls with RE/flex state APIs;
- preserve current source-location updates;
- regenerate or check in the generated scanner according to the build design.

The conversion should not introduce a parallel `RawToken` type unless there is
a concrete design reason to replace `LexedToken`.  The current `LexedToken`
contract already separates raw token recognition from parser-visible commit.
Virtual tokens should also remain outside source-token adjacency tracking:
operator adjacency is based on committed real source tokens, not inserted
`VOCURLY`, `SEMI`, or `VCCURLY` tokens.

## Parser-Error Layout Close

One layout behavior is still grammar-sensitive.  The grammar currently has a
`close: error` production that suppresses the speculative parse error and pops
a layout context.

For example:

```haskell
main = let x = 1 in x
```

The token `in` is not offside by indentation alone, but the parser needs the
`let` layout block to close before `in`.

GHC also documents case/where interactions where indentation alone is not
enough:

```haskell
f x = case x of
  True -> False
  where y = x + 1
```

The virtual close before `where` is discovered through parser feedback, not raw
indentation.

This should eventually become a narrow driver API, for example:

```c++
drv.close_layout_for_parse_error();
```

The method should:

- be used only from the `close: error` production;
- suppress the speculative parse error, preserving current diagnostics;
- verify that the top context is an implicit layout context if that invariant is
  available;
- pop exactly one implicit layout context;
- leave the current parser lookahead behavior unchanged.

This is not required before an ASCII-equivalent RE/flex raw-scanner conversion,
because the current parser action is independent of the scanner generator.  It
is still desirable cleanup because it makes layout context ownership explicit.

## Unicode Direction

The main long-term reason to use RE/flex is native Unicode handling.  Do not
mix Unicode behavior into the first scanner switch.

After the ASCII-equivalent conversion is stable, add Unicode support in small
steps:

- Unicode identifier starts and continuations;
- Unicode symbolic operator characters;
- Unicode whitespace if needed;
- tests for accepted and rejected Unicode lexemes;
- notes on known differences from GHC.

GHC's byte-class lexer strategy is useful background, but the preferred
BAli-Phy direction is to let RE/flex handle Unicode directly if it proves
practical.

## Implementation Sequence

Use a separate jj line of work for the RE/flex transition.  The older
`reflex-lexer-experiment` branch is a reference, not the base implementation.

Suggested commit sequence:

1. `docs: settle RE/flex build and license notes`
2. `build: add RE/flex dependency path`
3. `parser: introduce RE/flex scanner object lifetime`
4. `parser: generate Haskell raw scanner with RE/flex`
5. `parser: remove obsolete Flex-specific compatibility code`
6. `parser: route parse-error layout close through driver API`
7. `parser: add Unicode lexer coverage`
8. `parser: enable selected Unicode lexer behavior`

The parse-error layout cleanup can happen before or after the raw-scanner
conversion.  It should not block the initial RE/flex switch unless the scanner
object design naturally exposes a better layout ownership boundary.

## Test Plan

For each scanner conversion stage, run:

```sh
ninja -C build/gcc-16-debug-O src/bali-phy/bali-phy
meson test -C build/gcc-16-debug-O --num-processes 1 "bali-phy:bali-phy testsuite parse/18"
meson test -C build/gcc-16-debug-O --num-processes 1 "bali-phy:bali-phy 5d +A 50"
```

Also run focused tests for:

- `haskell/MultiWayIf/Layout`
- `haskell/MultiWayIf/Braces`
- `haskell/MultiWayIf/NoExtension`
- `haskell/Layout/IfNewlineCondition`
- `haskell/Lexer/Adjacency`
- `haskell/Records/Dot`
- recursive-do tests
- EOF-after-open-layout tests, if existing focused tests are available
- nested comments and pragmas, if existing focused tests are available

Before merging the maintained RE/flex conversion, run the broader parser and
Haskell tests.  If generated Haskell behavior changes, also run at least one
external `testiphy` pass.

## Risks

The main remaining risks are:

- platform-specific build failures from RE/flex integration;
- scanner state lifetime bugs across multiple parses;
- source-location drift during the scanner conversion;
- accidentally changing nested comment, pragma, or line pragma behavior;
- losing parser-error layout close behavior during later cleanup;
- treating Unicode differently from GHC in surprising ways;
- leaving behind compatibility workarounds after the generator switch.

The main semantic layout risks have already been reduced by moving virtual
token insertion into the driver wrapper before changing scanner generators.
